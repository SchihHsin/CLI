01 · 原生集成
无需切换上下文
零上下文切换终端原生
CLI 直接在开发者已有的终端中运行。无需浏览器、无需 GUI 窗口——AI 能力就在代码所在的地方，思维链路不中断。
02 · 可编排性
与工程工作流无缝融合
Shell 原生CI/CD 集成
天然支持管道、重定向和脚本化，可嵌入 Makefile、GitHub Actions、pre-commit hooks，真正成为工程流程的一部分。
03 · 效率优先
键盘驱动，毫秒启动
低摩擦极速响应
没有 GUI 渲染开销，没有 Electron 启动延迟。追求极致效率的开发者需要的是最短路径——从想法到执行，没有多余摩擦。
背景洞察 02 · 现状审查 OpenCode 的设计局限 — 设计优化的必要性
01 · 启动阶段
Banner 信息结构薄弱
启动时仅输出基础版本信息，缺乏工具状态、权限模式、模型上下文等关键启动态。开发者无法在启动瞬间建立完整的会话认知。
无权限模式指示
无模型 / Token 预算展示
02 · 表达意图
上下文管理能力缺失
输入框为纯文本，缺少 @file / @url 等结构化上下文注入机制，也没有 Context Chips 让开发者直观管理已添加的上下文范围。
无 @mention 上下文注入
无 Context Chips 可视化
03 · 等待执行
执行透明度不足
Tool Call 以纯文本流输出，缺乏结构化折叠展示；无清晰进度指示器；上下文压缩等关键事件静默发生，开发者无感知。
Tool Call 无结构化展示
上下文压缩无提示
04 · 审查确认
Diff 与审查体验简陋
代码变更以原始 diff 输出，无语法高亮；Accept / Reject 操作缺乏清晰的交互流；开发者需离开 CLI 去 IDE 中确认变更。
Diff 无语法着色
无 Accept / Reject 工作流
05 · 视觉 / 主题
视觉层次与品牌感弱
全局单色终端输出，错误、警告、成功等信息与普通文本视觉权重相同；缺乏主题系统，无品牌识别度。
无语义化颜色体系
缺乏视觉层次设计
06 · 多智能体
多 Agent 协作支持有限
多个 Agent 并行运行时无统一管理面板；子 Agent 状态、任务分配、进度跟踪均不可见，协作效率难以保障。
无 Agent 管理面板
子任务状态不透明
设计优化的必要性
上述六大维度的系统性局限贯穿用户完整操作路程——这不是局部打磨，而是需要从信息架构到交互设计的全链路重构。

核心洞察：框架选择决定了能做什么，而不只是长什么样
Warp 的 Block 体系把输出变成可操作结构单元，是 inline AI 追问的前提。Gemini 的 PTY 是唯一能运行所有交互式工具的方案。我们的方向：固定底部编辑器（成本最低的起点）+ 轻量 Block 概念（只对写入操作的输出加 badge，不依赖 GPU 渲染）+ 输入框内嵌 Context Chips 和费用角标。
Page Framework · 页面框架 CLI 页面框架 — 我们的方向
推荐页面框架：固定分离式 + 轻量 Block
以 Claude Code 的上下分离为基础，叠加 Warp 的 Block 可操作性概念，在保持兼容性的前提下最大化信息密度与可操作性。
1
固定底部输入区
输入框始终定位在底部 20% 区域，包含 Context Chips、模型切换与费用角标，输入位置永远稳定，降低用户视线跳跃成本。
2
轻量 Block 标注
仅对工具调用产生的写入操作输出加可折叠 badge，不依赖 GPU 渲染，兼容 SSH/tmux，同时提供类 Warp 的结构化感知。
3
顶部轻量状态栏（可选）
参考 Warp 的 status bar 概念，在顶部用单行显示当前模型、Token 用量与 Git branch，不占用主内容区。
4
渐进式升级路径
先落地固定分离式框架（与现有 Claude Code 完全一致），再逐步引入 Block badge，避免一次性重构风险。
总结：固定分离式框架成本最低、兼容性最强，是最合理的起点。轻量 Block 标注作为第二阶段叠加，可显著提升工具调用可读性，同时不破坏现有 PTY/SSH 场景。

Multi-Agent · Our Direction多智能体架构 — 我们的方向
Hub & Spoke 起步 + P2P 演进 + 子 Agent 上下文隔离
以 Hub & Spoke（Subagents）起步，实现成本最低；随产品成熟演进到 P2P Mailbox（Agent Teams 架构）。子 Agent 在独立上下文中执行，只向主 Agent 返回摘要，控制 token 成本。
01
Hub & Spoke 起步：主 Agent 委派子任务
主 Agent 负责规划和决策，子 Agent 负责执行（读文件、搜索、代码分析等探索性任务）。子 Agent 自动使用低成本模型（Haiku），完成后向主 Agent 返回摘要而非原始输出，避免主上下文被污染。
02
等待状态：暂停标记，不猜测继续
Agent 遇到权限不够或需要澄清时，自动暂停并标记橙色" 等待输入"，向用户发送通知。不会猜测"用户可能想要什么"而自动继续——这是"安全自主"的核心原则，避免雪球效应。
03
统一任务面板，人随时可介入任意 Agent
所有 Agent 任务在统一面板中可见，人可以点击任意任务查看详情或给新指令。不是"启动了就不管"，而是"随时可介入"。人是所有 Agent 的最终决策者，不是被排除在外的观察者。

Touchpoint · Our DirectionAgent 管理面板 — 我们的方向
统一任务面板 + 等待状态可视化 + 上下文隔离
多 Agent 有统一的任务管理面板，每个 Agent 有状态标记（运行中/等待/完成/失败）。Agent 遇到不确定时暂停等待而非猜测继续。子 Agent 在独立上下文中执行，主 Agent 只收摘要。
01
统一任务面板，状态一眼扫全局
所有 Agent 任务列在一个面板里，颜色和图标区分状态：运行中（绿色动画）、等待确认（橙色警告）、已完成（灰色）、失败（红色）。步骤进度条 + 耗时显示。可点击任意任务进入详情。
