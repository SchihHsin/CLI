架构模式Hub & Spoke 固定：Architect 规划 + Editor 执行，两 Agent 无法扩展Hub & Spoke（Subagents）+ P2P Mailbox（Agent Teams，实验性）两种架构Cloud Orchestration：事件触发，并发执行，统一管理面板，oz.warp.dev多模型混跑：Claude/Codex/Copilot 三种 Agent 在 VS Code Agent Sessions 面板统一管理Single Agent 云端 session，支持无头（headless）模式，GitHub Actions 集成，无多 Agent 协作
等待/卡住处理无处理机制，Agent 遇到问题会直接失败或无限等待Subagents 遇到权限不足时向主 Agent 返回错误，Teams 模式实验中等待输入的 Agent 自动标记橙色 warning，不卡死流程无专门的等待状态 UI，依赖 IDE 原有通知机制云端 session 可设置 timeout，本地无等待 UI
优势 最早的多模型实践，推理透明双模型 token 效率最佳 P2P mailbox 是行业最先进的协作架构Subagent 用 Haiku 执行，成本优化 统一管理面板，本地+云端混排等待状态黄色标记，不卡死Slack/GitHub/cron 事件触发 三种 Agent 统一管理，多模型混跑在 IDE 内，迁移成本最低 云端 session 持久化，可异步运行GitHub Actions 原生集成是开发工作流自然延伸
劣势 只有两个固定 Agent，无法横向扩展无统一管理面板 Agent Teams 仍实验性，需 env var 开启无统一管理 UI，并行状态分散 依赖 Warp 整体框架云端 Agent 中间过程透明度低 Agent 间暂无直接通信，仍是 Hub & Spoke在 IDE 侧边栏，非终端原生体验 无多 Agent 协作，仅单 Agent云端状态对用户不透明
总结最早实践，架构有局限架构最先进，UI 管理待完善云端调度最完善多模型混跑最方便—
最关键的 UX 问题：Agent 卡住时如何处理？Warp 的做法（黄色标记暂停等待，不猜测继续）是目前最安全的方案。推荐：统一任务面板 + 等待状态可视化 + P2P 通信减少主 Agent 瓶颈 + 子 Agent 上下文隔离（只向主 Agent 返回摘要，控制 token 成本）。
Multi-Agent · Our Direction多智能体 — 我们的设计方向
统一任务面板 + 等待状态可视化 + 上下文隔离
多 Agent 有统一的任务管理面板，每个 Agent 有状态标记（运行中/等待/完成/失败）。Agent 遇到不确定时暂停等待而非猜测继续。子 Agent 在独立上下文中执行，主 Agent 只收摘要。
01
统一任务面板，状态一眼扫全局
所有 Agent 任务列在一个面板里，颜色和图标区分状态：运行中（绿色动画）、等待确认（橙色警告）、已完成（灰色）、失败（红色）。步骤进度条 + 耗时显示。可点击任意任务进入详情。
02
等待状态：暂停标记，不猜测继续
Agent 遇到权限不够或需要澄清时，自动暂停并标记橙色" 等待输入"，发送通知给用户。用户处理完后 Agent 继续执行。"安全自主"的核心设计原则。
03
子 Agent 上下文隔离，主 Agent 只收摘要
子 Agent 在独立上下文窗口中执行（读文件、搜索等探索性任务），完成后只向主 Agent 返回摘要，避免大量中间输出污染主上下文，同时控制 token 成本。

Design Principles · Summary18 条核心设计原则
PHASE 01-02 · 启动 & 表达意图
零配置首次体验，30 秒内开始第一个任务
Banner 只展示：模型名 + 权限模式两个关键信息
新用户首次触发一次性引导，此后不再重复
输入框固定底部，首次加载即获焦点
模式 Badge 颜色区分（Agent 紫色 / Terminal 灰色），永远可见
@ 补全延迟 <100ms，Chips 超 3 个折叠为 "+N"
PHASE 03 · 等待执行（情绪最低谷）
Thinking 默认一行摘要，点击展开，完成后自动折叠
读取类 Tool call 自动折叠为计数，写入类展开并标色
高危操作自动触发 Plan Mode 先提案再执行
步骤计数（n/N）+ 整体进度条，双重反馈
失败高亮红色 + "让 AI 修复"快捷入口内嵌进度视图
Esc 中断按钮始终浮动可见，不需记快捷键
PHASE 03-06 · 确认 / 审查 / 迭代 / 收尾
弹窗颜色编码风险等级（蓝色读取 / 橙色写入 / 红色执行）
三档授权（仅此次/会话/永久）+ "修改后执行"第四选项
默认输出摘要行，变更 ≥ 3 文件自动建议 Review Panel
输出末尾恒有追问入口（✦），驱动自然迭代节奏
费用角标始终在 Chip 区可见，点击展开按任务明细
退出时展示会话摘要卡，强化完成感和费用意识
总体原则：让"AI 在做什么"始终可见，让"用户能做什么"始终触手可及。透明度不是噪音，而是信任的基础。自动化程度不是目标，掌控感才是。每一个设计决策都应该问：「这让用户更有掌控感了吗？」

Multi-Agent · Our Direction多智能体架构 — 我们的方向
Hub & Spoke 起步 + P2P 演进 + 子 Agent 上下文隔离
以 Hub & Spoke（Subagents）起步，实现成本最低；随产品成熟演进到 P2P Mailbox（Agent Teams 架构）。子 Agent 在独立上下文中执行，只向主 Agent 返回摘要，控制 token 成本。
01
Hub & Spoke 起步：主 Agent 委派子任务
主 Agent 负责规划和决策，子 Agent 负责执行（读文件、搜索、代码分析等探索性任务）。子 Agent 自动使用低成本模型（Haiku），完成后向主 Agent 返回摘要而非原始输出，避免主上下文被污染。
02
等待状态：暂停标记，不猜测继续
Agent 遇到权限不够或需要澄清时，自动暂停并标记橙色" 等待输入"，向用户发送通知。不会猜测"用户可能想要什么"而自动继续——这是"安全自主"的核心原则，避免雪球效应。
03
统一任务面板，人随时可介入任意 Agent
所有 Agent 任务在统一面板中可见，人可以点击任意任务查看详情或给新指令。不是"启动了就不管"，而是"随时可介入"。人是所有 Agent 的最终决策者，不是被排除在外的观察者。
