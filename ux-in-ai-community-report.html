01 The Shift

当用户不再来，
来的是它的代理。 The user no longer visits. Their agent does.

传统社区是一种"拉"的模型：网站坐在那里，等用户来。文档、下载、教程、课程， 每一份内容都假设有一双人类的眼睛在阅读、有一双人类的手在点击。 AI 改变了这件事——它不只是一种新工具，更是一个新的中介层。

今天，用户问 AI："帮我用这个框架做一个组件"，AI 在背后完成了过去用户要做的全部动作： 搜索文档、选定 API、写出代码、调试错误。用户从未访问社区，但社区被使用了。

这个变化的关键不在于"少了多少访问量"，而在于内容消费的回路被重新串接： 过去内容直接面向人，今天它必须先取悦一个中间译者。如果文档不能被 AI 准确摄取与转述， 那么这个社区对终端用户就等同于"不存在"。

Before / 旧范式
人 → 社区
  • 用户主动访问网站
  • 线性学习路径，按页阅读
  • 搜索 + 浏览 + 收藏
  • 内容为"页面"而设计
  • 成功 = 用户完成转化
↦
After / 新范式
人 → AI → 社区，或 Agent → 社区
  • AI 代理内容的检索、理解、执行
  • 路径由用户当下任务即时生成
  • 提问 + 协作 + 验证
  • 内容为"模块"和"技能"而设计
  • 成功 = 任务被正确完成（且可追溯）

如果 AI 是新的"读者"，那么文档就是新的"产品"。
UX 设计师的工作，正从"设计页面"变成"设计能被准确理解、引用、执行的知识。"

— Thesis One

02 Four Modes

让内容被 AI
更好地调用。 Making content callable, executable, and trustworthy.

"被 AI 调用"不是一个比喻，而是四种具体的技术与设计形态—— 它们之间没有冲突，社区通常需要同时铺设。

i.
结构化与机器可读
Machine-Readable Content

为内容铺一层"AI 可读层"：清晰语义结构、独立元数据、稳定 URL。 AI 读得准，转述才准。

llms.txtllms-full.txtOpenAPIschema.orgMarkdown
ii.
知识可执行化
Executable Knowledge / Skills

把教程从"叙事文章"重写为"技能包"：明确前提、步骤、产出与失败处理， 让 AI 代理可以直接执行而非"复述"。

SKILL.mdRecipe patternRunnable snippetsI/O contract
iii.
对话式接入
Conversational / RAG Layer

在文档之上提供问答接口，让 AI 直接基于权威内容回答用户问题， 并能跳转到原始证据。

RAG嵌入式 AI AssistantCitation / 引用回链
iv.
智能体接入
Agent Interface (MCP)

为社区资源提供 Agent 能直接调用的工具接口：搜索、下载、提交、订阅， 像 API 但面向 LLM 的行为协议。

MCP serverTools / ResourcesAuth flowPermissions
Important · 设计含义

这四层各对应一种 AI 与社区的关系。但它们不是"工程团队的事"—— 每一层都包含大量需要被设计的决策： 哪些内容暴露、以什么粒度暴露、如何呈现来源与置信度、出错时谁负责。 这正是 UX 设计师的领地。

03 New Roles · 核心

UX 设计师的六个新角色。 From page designer to knowledge systems designer.

在 AI 时代，UX 设计师的"工作"没有变少，但工作的形态变了。 下面六种角色，多数已经在头部社区与开发者平台中出现。 它们不是六个职位，而是一个设计师在一个项目里需要同时扮演的六个面。

i.

双重受众的信息架构师

为人类设计的页面 IA ↦ 为人与 AI 同时设计的知识 IA

过去 IA 关心"用户能不能找到这一页"。今天还要关心：AI 能不能不混淆这一段。 这意味着内容必须"原子化"：每个段落、代码块、概念定义都能脱离上下文独立成立，并附带充足的元数据。

一个长教程不再是 IA 的最小单位——一个"可独立被检索的知识片段"才是。

设计师要做的
  • 把文档拆解为可独立寻址的"内容原子"，每个原子有标题、主题、前提、产出
  • 制定双 IA：人类导航 IA + 机器消费 IA（如 llms.txt 结构）
  • 设计"内容卡片"组件，作为人与 AI 共享的基本单元
ii.

"AI 可摄取"的内容设计师

写得清楚就行 ↦ 既能让人读懂，也能让 AI 不误读

内容设计的新约束：避免反讽、避免上下文依赖、避免"见图说话"。 AI 不一定能看到图，不一定能识别玩笑，也不一定能从你的标题里推出语境。 需要把意思明示，而不是暗示。

这不是让写作变机械——而是让写作多一层"可被机器准确转述"的检查。 好的 AI-friendly 文档往往对人也更友好。

设计师要做的
  • 制定内容写作准则（Content Standards），明确禁止与推荐句式
  • 每篇文档前置 TL;DR + 适用场景 + 不适用场景
  • 所有图像、代码、视频都附等价文字说明，AI 才看得懂
  • 给 AI 写"使用说明"：哪些段落可引用，哪些是 example 不能照搬
iii.

对话式交互设计师

设计搜索框 ↦ 设计与社区"说话"的体验

当用户在文档里直接对话，体验问题变成了：怎么问才能问得清？答错了怎么办？多轮上下文怎么保持？ 这是新的交互模式，与表单、按钮、链接同样需要被精心设计。

设计师要做的
  • 设计提问引导（suggested prompts）、澄清流程、零结果状态
  • 设计"AI 回答 + 引用源"组件：答案与证据同框出现
  • 设计多轮对话的"上下文条"，让用户随时知道 AI 还记得什么
  • 设计明确的"我不知道"出口——比强答更重要
iv.

信任与透明度设计师

假定内容权威 ↦ 让用户随时能验证 AI 是否说错

AI 会出错，且会自信地出错。这不是技术问题——这是体验设计问题。 用户需要在不打断流程的前提下，知道：这句话从哪来？置信度有多高？我怎么核实？

设计师要做的
  • 设计"引用样式"：脚注式、悬浮式、行内式——选哪种取决于使用场景
  • 设计置信度的视觉语言（不是百分比，而是色阶、措辞、是否带保留）
  • 设计"一键到原文"的核实路径
  • 设计用户反馈纠错的入口与回流机制：用户改了之后会发生什么？
v.

学习体验设计师（个性化路径）

线性课程表 ↦ AI 协同生成的"此刻最适合你"

固定的"零基础→高级"学习路径在 AI 时代显得僵硬：用户的背景、目标、当下的代码项目千差万别。 AI 可以即时拼装一条当下最适合此用户的路径——但这条路径的形态、节奏、引导语，都需要被设计。

设计师要做的
  • 设计"项目导向的学习壳"：用户描述目标，AI 拆分子任务，每步关联社区原始内容
  • 设计"导师介入时机"——AI 何时只看、何时讲解、何时挑战学习者
  • 设计能力地图与已掌握状态的可视化（取代"完成了几节课"）
  • 保留人类讲师 / 社区高手的可见性，避免学习者孤立在与 AI 的二人世界里
vi.

智能体 UX 设计师（Agent UX）

设计用户的点击 ↦ 设计 Agent 的操作与人对它的监督

当 Agent 能代用户在社区里下载、订阅、提交工单，谁来决定它能做什么、什么时候要问一下人？ 这是一个全新的设计领域：人不在键盘前，但仍要保留控制感。

设计师要做的
  • 设计 Agent 权限的授权流程（一次性 / 长期 / 范围）
  • 设计"行动前确认"的语言与节奏——什么必须问、什么静默执行
  • 设计 Agent 工作日志的呈现：发生了什么、用了哪些资源、是否成功
  • 设计"撤销"与"接管"——人随时可以回到方向盘前

设计师不是被 AI 取代。设计师正在被请回到桌前—— 因为这些新角色，没有一个是工程师能独自决定的。

— Thesis Two

04 Action Framework

怎么开始：
四个阶段。 Diagnose · Restructure · Redesign · Measure.

这套框架适用于任何已有内容资产的社区——开发者文档、设计系统、教育平台、品牌知识库。 关键是从一小块内容开始试点，而不是一次重做全部。

Phase 01
诊断
Diagnose
  • 挑一个高频任务（如"快速上手"），观察用户实际通过 AI 完成它的过程
  • 用主流 AI 模型拷问现有文档，看它转述对不对
  • 盘点内容形态：哪些是叙事、哪些已是结构化
  • 研究：用户为什么不来社区了？是不是 AI 已经替代了入口
Phase 02
重构
Restructure
  • 把试点内容原子化，加元数据
  • 制定 llms.txt 与内容索引
  • 识别需要变成"可执行技能包"的教程
  • 设计统一的引用 / 来源样式
  • 评估是否需要建 MCP 接口
Phase 03
重设计
Redesign
  • 嵌入对话式入口（带引用）
  • 把"学习路径"页改为"AI 助你拟定路径"
  • 设计 Agent 行动确认 / 日志组件
  • 更新设计系统：新增"AI 卡片"族组件
  • 建立内容写作准则与培训
Phase 04
度量
Measure
  • 建立双指标体系（见 Ch.5）
  • 跟踪 AI 转述准确率（人工抽样审）
  • 跟踪用户的"二次验证率"——他们多频繁去核实
  • 跟踪 Agent 调用率与失败原因
  • 每季度迭代内容写作准则
设计师视角的提醒

四个阶段中，最容易被忽视的是 Phase 01 诊断。 不要在没看过用户实际"通过 AI 用你的内容"是什么样之前就开始重构。 像所有 UX 研究一样：先观察，再下笔。

05 Dual Metrics

既看人，也看机器。 A dual indicator system.

传统的 PV、UV、停留时间在 AI 中介的世界里失真。真正要看的是"任务是否完成"， 以及"内容是否被准确传递"——无论传递路径上有没有人。

类别
人类侧 / Human Side
AI 侧 / Machine Side
α
触达与任务完成
页面访问、搜索成功率、教程完成率、社区提问数
AI 调用与任务完成
内容被 AI 引用次数、AI 协助下的任务完成率、Agent 调用 MCP 成功率
β
理解与信任
阅读理解抽样、文档反馈"有用 / 无用"、回访率
转述准确率
AI 转述与原文一致性（人工抽样）、幻觉率、引用准确率
γ
满意与留存
NPS、社区活跃度、贡献者数量、二次回访
协作信任
用户对 AI 回答的接受率、二次验证频率、纠错提交率
δ
内容健康
过期文档比例、缺失链接、可读性评分
AI 友好度
原子化覆盖率、元数据完备度、MCP / llms.txt 暴露率
关键观察

随着 AI 转述准确率上升，人的访问量可能下降——这不一定是坏事。 如果用户通过 AI 顺利完成了任务，社区的价值被传递了，只是路径变了。 把"内容被准确使用"作为成功定义，比"页面被看到"更重要。

— Closing · Three Theses

设计师不会消失。
设计师在重新定义
自己的工作。

i.

从页面思维到知识系统思维

每个内容片段都是一个"独立运行的小程序"，必须自带元数据、前提、产出。 设计的最小单位不再是页面，而是可被任意组合调用的知识原子。

ii.

从转化漏斗到协作回路

成功不再是"用户走完路径"，而是"任务被正确完成"—— 无论是用户独立完成、与 AI 共同完成、还是 Agent 代为完成。设计要为这三种情形同时设计。

iii.

从权威输出到透明协作

信任不再来自"我们说了算"，而来自"你随时能看清证据、随时能纠正"。 这是 AI 时代体验设计的新基准线，也是设计师能做的最关键的贡献。

UX 设计师的下一站，不在屏幕里。
在内容与智能之间的那一层。
