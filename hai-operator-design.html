表达意图 · Stage 02 · 设计点 4/4 光标即上下文 — 停留在 TBE 代码行触发硬件分析
设计说明
代码感知 × 硬件映射
AI 持续监测光标在 TBE 代码中的位置。当工程师在 data_move、tik.Tensor、set_flag 等关键行停留超过 1.5 秒，自动浮出对应的昇腾硬件分析卡：内存层次访问效率、bank conflict 风险、CUBE 吞吐影响。
核心设计点
停顿检测阈值 1.5s，自动识别光标所在 TBE API 类型
data_move 行：显示 UB→L1→L2→HBM 搬运效率分析
Tensor 声明行：显示 UB 占用计算 + 剩余空间
非侵入式浮层，ESC 或移走光标即关闭
BEFORE过去的做法
tik.data_move(x_ub, x_gm, ...)
↓ 不确定搬运效率
查文档 → 手动计算带宽 → 反复 msprof 测
每次都要重新推算
AFTER新的设计
光标悬停在 data_move 行 1.5s，AI 自动计算该搬运操作的带宽效率和昇腾内存层次命中情况，给出优化提示。

定义完成 · Stage 03 · 设计点 1/1 Goal 命令 — 定义验收标准，AI 持续迭代到达成
设计说明
验收驱动的算子开发
工程师不再手动反复跑 msprof → 看数据 → 改代码 → 再跑。用 /goal 命令声明算子的验收条件（性能指标 + 精度门限 + 硬件兼容性），AI 自动迭代 TBE 实现，直到所有条件全部达成。
核心设计点
支持多维度验收条件：性能（CUBE 利用率）+ 精度（误差上限）+ 兼容性（910B/310P）
AI 自动生成多个 TBE 变体并并行评估
每轮迭代后输出进度报告，可随时暂停并查看当前最优版本
Goal 历史可复用：相似算子直接继承验收标准
BEFORE过去的做法
改 tile_m=64 → 跑 msprof → 38%
改 tile_m=128 → 跑 msprof → 51%
改 format → 跑验证 → 精度失败
回退 → 改 ub 分配 → ...
← 人工反复循环，~2天
AFTER新的设计
一次 /goal 声明验收条件，AI 自动迭代 TBE 变体 + 并行跑 msprof + 验证精度，达成即停止。

委托执行 · Stage 04 · 设计点 2/2 硬件看板 — 910B / 310P 多硬件并行测试一屏总览
设计说明
多硬件并行测试监控

910B 矩阵类算子需 FRACTAL_NZ，向量类用 ND。混用导致 EZ9999。
OOM · UB 超限

310P 精度差异
