# 竞品分析报告 Skill

本 Skill 帮助 Claude 系统性地完成一份完整的竞品分析报告，从信息搜集到可交付的设计方案。基于 AI CLI 设计研究项目（涵盖 Claude Code、Warp、Gemini CLI、Aider、Codex CLI 的 20 个交互触点分析）的完整实践经验提炼。

---

## 第一步：理解需求（2-3 个关键问题）

在开始工作之前，确认以下信息。如果对话中已经给出，直接使用，不要再问：

1. **分析领域**：什么产品类型？（AI 工具 / 设计软件 / 电商 / SaaS 等）
2. **分析目的**：做参考借鉴 / 找差异化机会 / 做设计决策 / 给领导汇报？
3. **产品范围**：有没有指定竞品？如果没有，AI 自主搜索确定主流产品（通常 4-6 个）。

> 如果需求很模糊，先开始工作，生成第一版后让用户反馈。不要等问题想清楚了再动。

---

## 工作流程

### PHASE 1 · 产品搜集与信息建立

**AI 自主执行，无需用户干预。**

```
1. web_search("领域 + 主流产品 + 年份")
2. web_search("产品名 + interface / UX / interaction")  
3. web_search("产品名 + comparison / review / teardown")
```

从搜索结果中提取：
- **产品范围**：确定 4-6 个主要竞品，兼顾头部 + 差异化
- **基本框架**：每个产品的定位、技术特点、目标用户
- **UI 特征**：截图、文字描述、可视化界面形态
- **独特数据点**：如"Warp 数据显示 96% 的 diff 直接 Accept"，这类数据在对比时极有价值

**输出**：内部知识库（不展示给用户），供后续分析调用。

---

### PHASE 2 · 用户旅程地图

**这是整个分析的骨架，所有触点必须从这里生长出来。**

#### 2.1 确定旅程阶段

通常 4-6 个阶段，覆盖完整使用周期：

```
示例（AI CLI）：启动 → 表达意图 → 等待执行 → 审查确认 → 迭代修正 → 收尾
示例（电商）：发现 → 浏览 → 决策 → 购买 → 等待 → 收货 → 售后
示例（SaaS）：注册 → 引导 → 核心任务 → 协作 → 数据导出 → 续订
```

#### 2.2 旅程地图结构（4 行）

| 行 | 内容 | 说明 |
|---|---|---|
| 用户行为 | 每阶段用户实际操作 | 具体动作，4-5 条，加图标 |
| 关键触点 | 可点击链接到对应分析 | 这些触点决定了后续的分析范围 |
| 情绪曲线 | SVG 折线，标出最高/最低点 | 情绪最低谷 = 最大设计机会 |
| 摩擦 + 机会 | 问题和改进方向 | 红色 = 摩擦，绿色 = 机会 |

> **重要**：不要放"用户想法"行。保持旅程图干净，4 行已经信息量足够。

#### 2.3 触点识别原则

从旅程地图中提取触点，**每个触点必须**：
- 对应旅程中的一个具体交互节点
- 是可以跨产品对比的 UX 决策点
- 有明确的"好/坏"判断维度

**维度拆分原则**（重要）：如果一个触点涉及多个不同维度，拆分为多个触点：
- ✗ 错误：输入框（位置 + 编辑能力 混在一起）
- ✓ 正确：输入框-位置/形态 | 输入框-编辑能力 — 分两个触点分析

---

### PHASE 3 · 触点对比分析

**每个触点一个页面，固定结构。**

#### 3.1 对比表格格式（核心）

```
列 = 产品（产品名为主标题，做法描述为副标题）
行 = 分析维度
```

**列标题格式**：
```html
<ph-name>Claude Code</ph-name>       ← 产品名（主）
<ph-sub>IDE 级多行编辑器</ph-sub>    ← 做法简述（副）
<badges>Anthropic</badges>
```

**行维度标准集**（根据触点特性选择 4-6 个）：
- 界面形态（必有，含 terminal mockup）
- 核心机制/做法描述
- 优势（绿色背景，+ 前缀）
- 劣势（红色背景，- 前缀）
- 一句话总结（紫色高亮行）

> 界面形态行必须有 terminal mockup 或 UI 截面图，不能只有文字。

#### 3.2 Terminal Mockup 规范

```html
<div class="mt">
  <div class="mtbar">
    <div class="mdt r/y/g"></div> × 3
    <span class="mtt">产品名 — 触点名</span>
  </div>
  <div class="mb">
    <!-- 用 ASCII art + 颜色类模拟实际 UI -->
    <!-- 颜色：.tg=绿 .tb=蓝 .tp=紫 .ty=黄 .tr=红 .tc=青 .tw=白 .td=灰 -->
    <!-- 标签：.tag .tg2/.tb3/.tp3/.ty3 -->
    <!-- 光标：.cur -->
  </div>
</div>
```

**Mockup 内容原则**：
- 展示该产品该触点的**最典型状态**
- 文字要具体，不要写"输入框"要写"❯ 重构 @src/auth.ts"
- 颜色选择要符合该产品的实际风格

#### 3.3 推荐方向页（每触点都有）

每个触点分析页之后，紧接一个"我们的方向"页：
- **左列**：核心理念（一句话）+ 3 条具体建议（每条有原因）
- **右列**：展示推荐方案的 terminal mockup（多个状态）

#### 3.4 核心张力公式

每个触点分析结尾必须有：
```
核心张力：[设计矛盾描述]（如"信息量 vs 界面清洁度"）
推荐：[具体行动建议，含数据或条件]
```

---

### PHASE 4 · 设计亮点提炼

在所有触点分析完成后，**跨触点综合**，提炼 6-10 个最重要的设计亮点。

**亮点页格式**：
- 大数字（01/02...）
- 亮点标题（一句话说清价值）
- 解释为什么这个设计决策重要（不是描述是什么）
- 右侧：多种状态的 mockup 对比

**亮点选择标准**：
- 解决了旅程中的关键摩擦点
- 与竞品有明显差异化
- 背后有可解释的设计原理

---

### PHASE 5 · 开发规格文档

为每个关键组件提供前端开发可直接使用的规格：

```typescript
interface ComponentName {
  // 属性列表（含类型、默认值、说明）
  prop: Type; // 说明
}

// 状态机
type Status = 'idle' | 'active' | 'done' | 'failed';

// 行为规格表格：
// 事件 | 触发条件 | 行为
```

**必须包含的规格**：
- Props/属性 + 类型 + 默认值
- 状态机（所有状态及转换条件）
- 事件响应规格
- 边界条件处理

```css
/* 每个楼层 */
.section {
  width: 100vw;
  height: 100vh;
  scroll-snap-align: start;
  scroll-snap-stop: always;
}

  scroll-snap-type: y mandatory;
```

**导航系统**：
- 两层 Tab：阶段（上） + 触点（下）

**图标系统**：
- 全部使用 SVG 线性图标（不用 emoji）
- 24x24 viewBox，1.5px stroke，round linecap

## 质量检查清单

在提交报告前自我审查：

### 结构完整性
- [ ] 用户旅程地图覆盖完整使用周期
- [ ] 旅程中的每个触点都有对应分析页
- [ ] 每个触点分析页都有推荐方向
- [ ] 所有产品列表一致（没有遗漏某个产品）

### 内容质量
- [ ] 列名是产品名（主）+ 做法（副），不是维度名
- [ ] 每个分析维度在同一触点内是同一个维度（不混用）
- [ ] 每个触点有 terminal mockup 展示
- [ ] 推荐方向有具体理由和条件，不只是"推荐 A 方案"

### 视觉呈现
- [ ] 没有 emoji（终端内的 ASCII 字符除外）
- [ ] 所有楼层 100vh，snap scroll 生效
- [ ] 两层导航 Tab 正确显示并可交互
- [ ] 字体大小可读（最小 12px 内容文字）

### 一致性
- [ ] 所有产品都分析了相同的触点集合
- [ ] Pro/Con 格式一致（+ 绿色 / - 红色，同一语法）
- [ ] 总结行格式一致（紫色背景，斜体简短描述）

## 常见问题与处理方式

### Q: 产品信息搜不到怎么办？
优先搜索：官方文档 / GitHub README / 深度评测文章 / YouTube demo。实在找不到的信息，在分析中标注"无公开信息"，不要猜测。

### Q: 某个产品某个触点没有这个功能？
这本身就是分析结论。在那个格子里写"无此功能"，在劣势行标注这个缺失，可以在总结行写"缺失该功能，需第三方补充"。

### Q: 触点维度划分不确定？
判断标准：不同产品在这个维度上是否做了不同的设计决策？如果答案是否，说明维度太细；如果不同产品的答案无法用同一个维度描述，说明需要拆分。

### Q: 多少个触点合适？
根据研究深度：
- 快速竞品调研：5-8 个关键触点
- 完整 UX 研究：15-25 个触点
- 本 Skill 验证的案例：20 个触点，覆盖完整用户旅程

### Q: 要不要包含"我们的方向"？
如果只是内部研究参考，可以省略推荐方向，只做对比分析。如果要形成设计决策依据，推荐方向是必须的——否则报告只是"我们知道别人怎么做"，没有"我们应该怎么做"。

## 协作模式参考

基于本次 AI CLI 分析项目总结的高效协作模式：

| 模式 | Human 做什么 | AI 做什么 |
|---|---|---|
| 维度拆解 | 指出"这两个不是同一维度，原因是…" | 理解原则，找出所有同类问题一起拆分 |
| 批量修正 | "所有 emoji 改成 SVG 线性图标" | 审计清单 → 建图标库 → 脚本批量替换 → 验证 |
| 内容填补 | "对应旅程地图补全缺失触点" | 自主 audit 列表 → 发现缺口 → 自动补充 |
| 结构重构 | "拆成多个文件，用跨文件导航" | 设计文件架构 → 实现导航系统 → 维护一致性 |
| 合成升华 | "基于所有分析，输出设计方案" | 跨触点综合 → 提炼亮点 → 生成规格文档 |

**最重要的原则**：给"为什么"比给"怎么做"更有价值。AI 理解了判断原则，会举一反三处理所有同类问题。
