# AI 时代设计提案 Skill

基于 AI 时代开发者工具设计方案项目（2026）提炼。该项目覆盖 AI 辅助开发工作流的完整用户旅程（7 个阶段 × 6 个维度），提炼出 11 个设计点，每个设计点均包含研究依据、Before/After 对比和动态 Mockup，最终形成可交付的演示文稿。

---

## 第一步：理解需求（最多 3 个问题）

开始前确认，对话中已说清楚的跳过：

1. **设计领域**：是什么产品 / 工具 / 工作流？（AI 编程工具 / AI 写作工具 / 团队协作工具 / 其他）
2. **目标用户**：谁在用？典型使用场景是什么？（开发者 / 设计师 / 运营 / 通用知识工作者）
3. **提案目的**：内部对齐 / 评审汇报 / 对外展示 / 融资路演？

> 如果信息不全，先根据已知信息开始 Phase 1，研究过程中会自动补全。

---

## Phase 1 · 用户行为研究

**目标**：理解目标用户在使用 AI 工具时，实际上在做什么、遇到什么、感受如何。

### 1.1 研究框架

调研时围绕以下问题搜集材料：

```
- 用户完成一次典型任务的完整流程是什么？（从启动到收尾）
- 在哪些环节用户最频繁地出错、放弃、或切换工具？
- 当前的 AI 工具在哪里给了用户惊喜？在哪里让用户沮丧？
- 有没有用户已经自发发展出的"变通做法"？（这些往往是最大的设计机会）
```

### 1.2 信息搜集来源（优先级排序）

1. 用户访谈 / 可用性测试记录
2. 真实使用的工具（Claude Code、Cursor、Copilot 等）的 changelog / 官方博客
3. 用户社区（Reddit r/ClaudeAI, GitHub Issues, HN 讨论）
4. 竞品的 UX teardown 文章 / YouTube demo
5. 如果都没有：用 web_search 搜索 "工具名 + pain points / frustration / workflow"

### 1.3 输出：原始痛点清单

整理成结构化列表，每条包含：
- 触发场景（什么情况下）
- 用户行为（用户做了什么）
- 情绪状态（沮丧 / 困惑 / 焦虑 / 满意）
- 估计频率（高 / 中 / 低）

## Phase 2 · 用户旅程地图（7阶段 × 6维度）

**这是整个设计提案的骨架，所有设计点都必须从这里生长出来。**

### 2.1 定义旅程阶段

以 AI 辅助工作流为例（可根据领域调整）：

| 阶段 | 典型 AI 开发场景 | 通用模板 |
|---|---|---|
| 01 | 启动 / 建立上下文 | 启动 |
| 02 | 表达意图 / 下达任务 | 输入 |
| 03 | 委托执行 / 等待 | 执行 |
| 04 | 监控进度 / 多任务 | 监控 |
| 05 | 出错恢复 / 调试 | 异常处理 |
| 06 | 审核结果 / 验收 | 审核 |
| 07 | 收尾沉淀 / 知识留存 | 沉淀 |

> 阶段数量通常 5-8 个，覆盖完整使用周期，不能只写中间的"核心操作"。

### 2.2 六维度分析

每个阶段必须分析以下 6 个维度：

| 维度 | 内容 | 说明 |
|---|---|---|
| 阶段 | 阶段名称 + 简短描述 | 用户在哪里 |
| 触点 | 具体的交互界面 / 功能 | 用户触碰了什么 |
| 行为 | 用户实际执行的动作 | 用户做了什么 |
| 情绪 | 0-100 情绪分值 + 描述 | 用户感受如何（用于绘制曲线） |
| 痛点 | 当前体验中的问题 | 用户在哪里受挫 |
| 机会点 | 可改善的设计方向 | 我们能做什么 |

### 2.3 情绪曲线解读原则

- **情绪最低点** = 最大的设计机会，优先提炼为设计点
- **情绪突然下跌处** = 体验断层，通常是当前工具的设计盲区
- **情绪高点** = 验证有效的交互模式，新设计应保留或强化

### 2.4 从旅程到设计点

从旅程地图中圈出所有痛点，按以下标准筛选为正式设计点：

```
筛选标准（同时满足以下 3 条）：
✓ 影响频率高（用户经常遇到）
✓ 影响程度深（情绪分值低 / 落差大）
✓ 有明确的解决方向（不是纯技术限制问题）
```

## Phase 3 · 设计点提炼

### 3.1 组织成模块

将所有设计点按所属旅程阶段分组，形成模块结构。

以本项目为例：

| 模块 | 覆盖旅程阶段 | 包含设计点 |
|---|---|---|
| Module 1 · 监控进度 | 04 | Agent View / 并行看板 |
| Module 2 · 表达意图 | 02 | 光标上下文 / 手势Prompt / 语音×光标 / 光标伴侣 |
| Module 3 · 委托执行 | 03/05 | 智能Fix / Goal命令 |
| Module 4 · 记忆与沉淀 | 01/07 | 记忆启动 / 团队记忆库 |
| Module 5 · 审核结果 | 06 | 结果审核 / 意图对齐 |

> 每个模块 1-3 个设计点，不超过 4 个。模块数量 4-6 个，总设计点 8-15 个。

### 3.2 每个设计点的结构

每个设计点必须包含以下 4 个层次：

**① 洞察（一句话说清楚发现了什么问题）**
```
格式：[当前行为] + [导致的问题] + [原因]
示例："开发者需要在多个终端和文件之间手动切换，
      导致上下文频繁断裂，因为没有统一的多 Agent 状态视图。"
```

**② 设计说明（解释新设计的核心逻辑）**
```
格式：[新的机制/界面] + [如何解决问题] + [对用户的价值]
通常 3-4 句话，不超过 60 字。
```

**③ 核心设计点（3-4 条具体的设计决策）**
```
格式：[设计要素名称]：[具体做法] + [为什么这样做]
示例：
- 分组会话列表：按"需确认/运行中/已完成"三态分组，减少认知扫描成本
- Context 健康条：每行显示上下文使用率，>75% 自动预警，避免超限截断
```

**④ Before/After 对比**
```
Before（过去的做法）：视觉 Mockup 还原"没有新设计时用户怎么做"
  → 终端操作类：多个终端窗口并排 + 手动切换的箭头
  → 信息散落类：Slack / Notion / 邮件 / 脑中记忆的图标散布图
  → 操作繁琐类：多步骤流程箭头图（步骤 1→2→3→4）

After（新的设计）：简洁文字说明，真实 Mockup 在右侧 dp-right 展示
```

## Phase 4 · 叙事结构设计

### 4.1 提案的整体叙事弧

```
第一幕：背景（为什么现在是转折点）
  → 信号：AI 正在改变开发方式（数据 / 现象 / 行业动态）
  → 现状问题：当前工具的设计还停留在"前 AI 时代"

第二幕：旅程（用户实际上在经历什么）
  → 7阶段旅程地图：完整呈现用户体验全貌
  → 情绪曲线：让观众感受到痛点的真实性

第三幕：设计（我们发现了什么机会，提出什么方案）
  → 按模块展开每个设计点
  → 每个设计点：洞察 → 方案 → Before/After → Mockup 演示

第四幕：总结（一个方向，多个入口）
  → 核心主张：AI 工具设计的本质是让 AI 消失在工作流中
  → 设计系统全览（可选）
```

### 4.2 设计点的呈现顺序原则

- 先放视觉冲击力最强的设计点（Agent View / 光标追踪 等有动画的）
- 前 3 个设计点决定观众是否继续看下去
- 相关设计点相邻（同一模块的放在一起）
- 最后收尾的设计点应该有宏观感（记忆 / 沉淀 / 团队协作）

### 4.3 Sub-label 格式规范

每个设计 section 的 sub-label 需说清楚位置感：

```
格式：[模块名称] · Stage [XX] · 设计点 [N/M]
示例："表达意图 · Stage 02 · 设计点 2/3"
```

## Phase 5 · Mockup 设计原则

### 5.1 Mockup 的核心目标

**不是为了好看，而是为了让观众在 3 秒内理解设计意图。**

- 静态 Mockup：说明布局、信息架构、视觉层级
- 动态 Mockup（JS 动画）：演示交互流程、状态变化、时序关系

### 5.2 什么时候用动态 Mockup

| 设计点类型 | 推荐形式 |
|---|---|
| 状态列表 / 多会话管理 | JS 渲染列表 + 点击交互 |
| 光标追踪 / 位置感知 | SVG 飞行动画 + IntersectionObserver |
| 实时数据 / 进度 | CSS progress bar + transition |
| 代码 / 终端输出 | 纯 HTML 终端模拟（静态即可） |
| 信息架构 / 布局 | 静态 Mockup |

### 5.3 Mockup 视觉层级

每个 Mockup 必须有一个视觉焦点：
- 一个高亮的按钮 / 元素
- 或一个动态的状态变化
- 观众视线不能同时被三处以上吸引

### 5.4 Before Mockup 常见模式

根据设计点类型选择：

| 问题类型 | Before Mockup 模式 |
|---|---|
| 信息分散 | 图标散布图（多个来源飘散在画面中）|
| 操作繁琐 | 多步骤箭头流程图 |
| 界面割裂 | 多窗口并排 + 手动切换标注 |
| 上下文丢失 | 两个 Session 都在重复输入同样信息 |
| 反馈不明 | 只有"✓ completed"但开发者困惑的气泡 |
| 信息黑箱 | 黑箱图 + 只有 Accept/Reject 的选择 |

## 质量检查清单

### 内容完整性
- [ ] 用户旅程覆盖完整使用周期（有头有尾，不只有核心操作）
- [ ] 每个设计点都能追溯到旅程中的具体痛点
- [ ] 每个设计点有：洞察 → 设计说明 → 核心设计点 → Before/After
- [ ] Before 面板是视觉 Mockup，不是纯文字

### 叙事质量
- [ ] 第一幕（背景）能让不了解领域的人快速理解"为什么这很重要"
- [ ] 设计点之间有逻辑连贯性（不是随机罗列）
- [ ] 总结页能用一句话说清楚整个提案的核心主张
- [ ] Sub-label 的阶段编号与旅程地图阶段一致

### Mockup 质量
- [ ] 核心设计点有动态演示（不全是静态截图）
- [ ] 每个 Mockup 有一个明确的视觉焦点
- [ ] Before Mockup 能让观众在 3 秒内看懂"之前的问题"

## 参考：本项目的核心设计主张

> "AI 工具的设计目标不是'让 AI 更强大'，而是'让 AI 消失在工作流中'——用户不应该感觉自己在'使用 AI'，而是感觉工作本身变得更顺畅了。"

这个主张来自对 7 个旅程阶段的综合观察：每一个痛点，本质上都是 AI 的存在打断了用户的心流（需要手动切换、需要重复解释、需要猜测 AI 在做什么）。因此每个设计点的方向，都是在消除这种打断感。
